## Session 3: Unify chrome and content on one neutral surface system

**Status:** Implementation complete; PR CI pending

### Root cause

- Session 1 (#248) neutralized **Layer 2 only**: content cards moved from a teal wash to opaque `controlBackgroundColor`.
- Every **Layer 1** chrome tint was left untouched ‚Äî `Surface.companionTint` stayed teal, and `MeterBarDetailBackground` still painted a `codexAccent`/`appAccent` `LinearGradient` over its material.
- Result: blue/teal chrome wrapped around gray cards ‚Äî two competing surface systems in one window.
- Compounding: in dark mode `controlBackgroundColor` (~#1E1E1E) is *darker* than `.regularMaterial`, so opaque cards read as recessed slabs rather than lifted content.

### What was done

- `Surface.companionTint`: teal ‚Üí luminance-matched graphite, so the popover and detail-panel glass carry no product hue.
- `MeterBarDetailBackground`: deleted the accent `LinearGradient`, leaving plain `.regularMaterial`. One edit de-blues both the main window and the fixed-size Settings window (the latter uses the same background).
- `Surface.content`: opaque `controlBackgroundColor` ‚Üí neutral translucent white lift (0.55 light / 0.08 dark; 0.85 / 0.14 under Increase Contrast) so cards lift off the chrome instead of punching an opaque hole in it.
- Tests first: relaxed and renamed the content-surface regression test, and added `testChromeTintIsHueNeutral` ‚Äî the invariant whose absence let this ship.
- `.agents/DESIGN.md`: documented the content surface accurately and added an explicit "never tint one layer while the other stays neutral" rule.

### Decisions

- **No `contentOpaqueFallback` reintroduced.** A flat alpha wash is not vibrancy, so a translucent fill over an opaque backdrop already satisfies Reduce Transparency; #248's token simplification stands.
- **`sidebarMenuTint` left unchanged.** It is a control/selection accent, not a surface fill; retinting it would be scope creep on a surface-color complaint.
- **`DashboardCostCards` shimmer left unchanged.** It is a transient loading skeleton, the documented one-off decorative gradient, not a surface.
- `SocialShareCard` verified unaffected ‚Äî it never routes through `meterBarCardSurface` and paints with its own opaque literals, so exported PNGs are unchanged.

### Verification

- Local builds, tests, and typechecks skipped under the MacBook verification policy; PR CI is the execution gate.
- Every card consumer (`ProviderStatusViews`, `DashboardCard`, `MenuBarDetailPanel`, `Components/LimitRow`) inherits the new fill through `meterBarCardSurface` ‚Äî no per-call-site edits needed.
